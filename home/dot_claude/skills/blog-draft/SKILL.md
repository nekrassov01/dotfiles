# blog-draft

筆者の技術記事の文体で、日本語のマークダウン技術記事を書く、または推敲するためのスキル。実記事から抽出した文体パターンを再現し、そのまま手直しして公開できる「叩き台」を出す。出力は素の Markdown（frontmatter なし）。

## いつ使うか

- `/blog-draft <内容>` で呼ばれたとき（最優先）。`<内容>` が題材
- 技術ブログ記事のドラフト作成や推敲を、筆者の文体で行うとき
- 題材は AWS、Go、AWS CDK、Amazon Bedrock、生成 AI、IaC、自作 OSS ツールが中心
- 種類は「ツール紹介」「TIPS」「実装解説」「〜してみた（実験）」「検証・選定」のいずれか

## 基本方針（他のすべてに優先）

出力は、筆者の実文体よりも**わずかに堅め**に寄せる。実コーパスは「本編は堅く正確、地の文や括弧、所感で砕ける」二層構造だが、このスキルでは砕ける側を一段抑える。安易な口語が品位を下げやすく、堅めの方が技術記事として再利用しやすいため。

「堅め」とは口語の砕けを抑えることであって、筆者の声を消すことではない。次の区別が最重要。

- 抑える（砕け）: 安易な口語副詞、感情記号の連打、自虐の過多、本編に持ち込む雑談、共感や問いかけの多用
- 残す（声）: 一人称の判断（「自分はこうした」）、実体験や案件の背景、率直な迷いや限界の自認、謙遜や免責

一人称の判断や経験談まで消して一般論の仕様比較に陥ると「堅すぎ」で失敗。狙いは「本人だと言われれば信じるが、いつもよりきちんとしている」程度。お役所文書にはしない。

## 入力の扱い（`/blog-draft <内容>`）

`<内容>` は記事の題材で、粒度はまちまち。

- 一言テーマ（例「自作した CLI の紹介」）→ そのまま構成を起こして書く
- 箇条書きメモ、実装ログ、コマンド履歴 → 記事の流れに再構成する。順序は読者目線で組み替えてよい
- URL やリポジトリ、ファイルパスが渡されたら、可能な範囲で中身を読んで反映する
- 不足情報があっても**質問で止めない**。叩き台を出してから TODO で確認点を残す。ただし題材が一言すぎて方向が定まらないときだけ、記事種別（紹介、TIPS、解説、実験、検証）を一つ確認する

## 構成

冒頭は必ず `## はじめに`、末尾は必ず `## おわりに`。間は記事種別で変える。型は筆者の癖を再現する目安で、題材に合わせて足し引きしてよい。見出しは `##` を主、`###` を従とする 2 階層を基本にする。

- ツール紹介: はじめに / 概要 / インストール / コマンドとオプション / 使用例（基本→フィルタ→出力形式と段階的に、番号付きケース可）/ 注意点 / 今後の展望 / おわりに
- TIPS: はじめに / 使いかた / TIPS（各項目を `###` で並べ、問題提起→コード→解説。見出しは動詞止め可「〜を作る」）/ おわりに
- 実装解説: はじめに / 概要 / 実装（`###` で分割）/ 汎用化してみた / おわりに
- 〜してみた: はじめに / 構成 / 実装やコード / 動かしてみる / おわりに
- 検証・選定: はじめに / 概要 /（比較軸を見出しに）/ Pros/Cons / 採用理由 / おわりに

### 結論先出し（推奨オプション）

設計判断や結論が主題の記事では、`## はじめに` の直後に `## 最初に結論` を置き、結論を箇条書きで先出しする。読者が結論から逆に読めるようにする。

### 機能的な見出し

見出しは叙述調より機能的なラベルに寄せる。比較記事なら評価軸そのものを見出しにする（「導入容易性」「移行容易性」「金額」「クォータ」など）。課題は番号を振る（`### 課題 1:`）。同種の要素は反復構造で並べ、対称性を保つ。

### 書き出し（はじめに）

次のいずれかから入り、末尾を「今回は〜紹介します／してみました」のスコープ宣言に着地させる。動機、前提、過去記事参照を主にし、共感や問いかけは書き出しで一度程度に抑える。

- 過去記事参照（シリーズもので有効）: 「以下の記事で <前提となる内容> を紹介しました。」
- 動機・機会: 「<対象> を <扱う> 機会がありましたので紹介します。」
- 前提・状況提示: 「ひきつづき <題材> を触っています。今回は〜」
- 共感・問いかけ（多用しない）: 「<サービス> って <あるある> しますよね。」
- 予防線（任意）: 専門外の題材では「私は <領域> に詳しくないので、不備があるかもしれません」を一言添える
- クロスリンクは「はじめに」に置く方が筆者らしい

### 結び（おわりに）

核は 2 要素。要約一文と、前向きな次の一手。任意で読者への促しや所感を足す。

- 要約: 「<やったこと> について紹介しました。」
- 次の一手: 「<次の構想> してみたいと思います。」「引き続き手を動かしてみたいと思います。」
- 読者への促し（任意）: 「ぜひ使ってみてください。」「ぜひ検討してみてください。」
- 所感（レポート、検証系）: 学びや枯れ具合の評価で締める。「サーバーレスを選ぶメリットが増していると感じます。」

## 声とトーン

- ですます調で通す。常体は箇条書き内の体言止めに限る
- 自信と謙虚さの両立。「自分の場合はこうした」であって「これが唯一の正解」ではない
- 断定は確度で書き分ける。検証済みは言い切り、推測は「おそらく」「〜かもしれません」、未確認は「(未検証)」と明示する。ヘッジは機械的に連発しない
- 自虐や謙遜は筆者の指紋。置き場所を絞る（はじめにの予防線、おわりにの所感、丸括弧の短い補足）。本編の技術説明は淡々と正確に保つ
  - 例「私は〜が門外漢なので、不備があるかもしれません」「たいしたツールではないのですが」「間違っていたらごめんなさい」
- 丸括弧での短い補足や自虐が癖（半角スペース+半角括弧「 (後述します)」）。多用はしない
- ときどきメタな所感を一言（趣味と業務のスキル循環、「枯れてきた」という評価など）。結びに置く

## 本編の進め方

- 問題を率直に言語化してから解く。課題が主題なら `## 課題` を立てて箇条書きで列挙し、「そこで〜」で解法に移る
- 手順は明示的に区切る。「概ね以下のような流れです。」と全体像を箇条書きで示してから、各ステップにコードと一文の説明を置く
- コードは省略せず提示し、全体提示→部分再掲で解説する。要点は「重要なのは〜という点です」「肝は〜」で先出しする
- トレードオフは対称構造で書く。`## Pros/Cons` →`### Pros`／`### Cons`、または「メリット/デメリット」。片側が空でも「なし」と明示して対称性を崩さない
- 主張には根拠を添える（数値、クォータ、料金、出典リンク、公式ドキュメントの引用）
- 検証・選定記事では一般論の比較で終わらせない。「自分のケースではどの軸が効いたか」を一人称で一段添える。実体験が無ければ `（TODO: 自分のケースの判断材料を補足）` を置く
- 限界や適用条件を率直に書く。「本番で使うには〜のため、〜を待つのが安全です。」「網羅的なテストではないことに注意してください。」
- 同型のリード文を機械的に反復しない。「〜は次のとおりです」「〜という軸です」を各節頭に均等配置すると生成物特有の単調さが出る。導入の語と文長を散らし、節によっては前置きなしで本題に入る

## コード、図、表の見せ方

- コードブロックは**正確な言語指定**で syntax highlight する
- インラインコメントは日本語で意図を書く。要点は「// ここに注目」「// コレ」で強調する。長い実装では「// Step 1:」〜「// Step N:」の番号コメントを振ってもよい
- コマンドはカジュアルすぎない導入文の後にブロック。「以下のコマンドを実行します。」「Mac であれば homebrew でインストールできます。」
- コマンド出力やコンソール結果は monospace でそのまま見せる。実行例はプロンプト（`$` / `PS>`）付きで入力と出力を同じブロックに置く
- オプション、パラメータ、比較、注釈付きの一覧は表にする。セル内改行は `<br>`
- 図（フロー、アーキテクチャ、シーケンス、状態遷移）は mermaid で描く。独自の ASCII アートや罫線作図はしない（AI 臭が強い）。手描き画像へ差し替える前提なら `（TODO: 図を画像化）` を添える
- 図に文章を詰め込まない。ノードやラベルは識別できる最小限の名詞にとどめ、説明や条件は図の外（本文、表、キャプション）に出す
- ディレクトリ構成は tree（罫線ツリー、途中省略は `:`）で示す。各要素に補足注釈が必要なら tree を避けて表にする。`secretlint/{a, b}` のようなインライン注釈は足さない
- UI を伴う検証はスクリーンショットを置き、キャプションで結果を説明する。画像が無ければ `（TODO: スクリーンショット）` と置く

## 書式ルール

- ですます調を徹底する。本文に絵文字は使わない
- 太字は装飾でなく強調に限る（キーワードや警告などの要点に絞る）。斜体は使わない。技術用語とコードは backtick
- 装飾でなく構造（見出し、表、リスト、コード）で読ませる
- 中黒（・）を多用しない。語の並列は読点（、）や と / や に寄せる。特に 3 つ以上の並列は読点で区切る
- 煽り表現（「最強」「完全攻略」「決定版」）、宣伝臭の強い締めは避ける。トーンは「自分はこう使っている、よければどうぞ」
- 水増ししない。同じ内容を切り口を変えて繰り返さない
- 出自を明かさない。AI で書いた、生成したという断り書きやメタ言及を入れない。筆者本人が書いた一本の記事として仕上げる

校正ツール（後述）に通す前提で、次も最初から守る。

- 一文は 150 字以内を目安にする。極端な長文だけ分割する
- チェックリストや列挙の箇条書きは体言止めを基本にする。本文は ですます のままでよいが、箇条書きを ですます で終えない
- 「することができます」のような冗長表現は簡潔形（「できます」）にする
- 日本語と半角英数字の境界には半角スペースを入れる（`git 2.54 の` のように）
- 行頭が `#` で始まるタグ行は、先頭を `\#` にエスケープする（markdownlint の見出し誤認を防ぐ）
- 「！！」「。。。」「？」などの強調記号は使わない

## 言い回しパレット

筆者らしさの核として残す表現。

- 謙遜や免責: 「たいしたツールではないのですが」「私が説明するまでもないですが」「間違っていたらごめんなさい」「破壊的変更が入る可能性はありますが、よければ使ってください」
- 問いかけ（書き出しで一度）: 「〜しますよね」「〜と思ったことはありますか」
- 前向き: 「近いうちに試してみたいと思います」「今後はやはり〜してみたいと思います」「引き続き手を動かしてみたいと思います」
- 評価や所感: 「いい循環だと実感しています」「枯れてきたということかもしれません」

口語副詞（「ざっくり」「いい感じに」「サクッと」「ゴリゴリ」）は本人も使うが、堅め方針では抑制する。

## 叩き台としての扱い

出力は「手直しして公開する前提のドラフト」。完成品を装って事実を捏造しない。一方で、TODO ばかりにして本文を空にもしない。叩き台でも一本の記事として読める状態を優先する。

- 記事の核（ツール紹介の実コマンド出力、〜してみたや実装解説の動作確認、検証・選定の比較数値）は TODO だけで済ませない。**もっともらしいダミー値や想定出力を本文に置いたうえで** `（TODO: 実際の値に差し替え）` と注記する
- 公知の事実（AWS の公式上限値、料金体系、一般的なコマンド書式など確度の高い既知値）は断定して埋めてよい。曖昧なら確認する
- 実装解説や〜してみた のコードは、少なくとも一貫して動作する形にする。未解決の設計は本筋に残さず、おわりにの「次の一手」へ回す。動かないコードを「実装してみた」として出さない
- 未検証の固有数値やベンチマーク、実環境の挙動は `（TODO: 要確認）`、筆者しか知らない動機や経緯は `（TODO: ○○を補足）`、不明なリンク先は `（TODO: リンク）` にする
- 顧客名や案件名、社内固有名詞は出さない。必要なら「ある案件で」等にぼかす
- 記事末に「## 確認してほしい点（公開前に削除）」を付け、筆者の判断が必要な重要項目を**簡潔に（5 項目程度を上限）**箇条書きで残す。TODO の全件転記はしない

## 仕上げ

草稿ができたら、subagent に批評させて仕上げる。記事と本スキルを渡し、筆者の文体と次のチェック項目に照らして逸脱を探させ、指摘を反映する。subagent が問題なしと判断するまで繰り返す。独立した視点で、生成時に見落とす逸脱（堅すぎ、砕けすぎ、一人称の判断の欠落）を拾う。

- [ ] `## はじめに` で始まり `## おわりに` で終わる。結論が主題なら `## 最初に結論` を置く
- [ ] 書き出しが 動機、前提、過去記事参照、問いかけ のいずれかで「今回は〜」に着地し、共感や問いかけは書き出しで一度程度
- [ ] ですます調で一貫し、自虐や謙遜が はじめに、おわりに、括弧 に局所化され、本編は淡々と正確
- [ ] 一般論に終始せず、一人称の判断や経験が本編に残っている
- [ ] トレードオフが対称構造（片側空でも明示）で書かれ、主張に根拠が添えてある
- [ ] 断定が確度で書き分けられ、未検証は `(未検証)` か TODO。本文の表、コード、動作確認の過半が TODO で埋まっていない
- [ ] 実装や〜してみた のコードは動く形で、正確な言語タグと日本語インラインコメントがある
- [ ] 図は mermaid、構成は tree か表。独自の ASCII 作図やツリーへのインライン注釈をしていない
- [ ] 本文の絵文字ゼロ、太字は強調限定、中黒を多用していない、煽りや宣伝臭、AI 出自の断り書きがない

仕上げに markdownlint と textlint を通す。`--fix` で直せる指摘を自動修正し、残りを直す。指摘がなくなるまで繰り返す。

```bash
markdownlint-cli2 --fix --config ~/.config/markdownlint/.markdownlint-cli2.yaml <file>
textlint --fix --config ~/.config/textlint/.textlintrc.json <file>
```
